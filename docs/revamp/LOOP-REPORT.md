Ciclo 2.2 alinha o contrato `AgentRuntime`: `PermissionDecision::Deny` agora carrega o mesmo `DenyScope` (espelho local documentado — `bastion-agent-runtime` não depende de `bastion-types`); com `Turn`, o adapter cancela a task graciosamente após negar, fechando o gap de roteamento por tool alternativa também no lado do adapter, não só no kernel. Testado no fake (`deny_turn_scope_cancels_the_task`, `tests/agent_runtime_conformance.rs`) e implementado em `CodexAppServerRuntime::respond_permission` (`crates/bastion-agent-runtime/src/codex.rs`).
6. **[Ciclo 2.4 — achados na prática, não bloqueantes]** Quatro furos/impasses reais descobertos integrando `BackendProfile`/modo 2/modo 3 e rodando A-06/A-07 ao vivo (detalhe completo em `docs/revamp/A-06-A-07-live.md`):
   - ~~**(a) Aprovação síncrona não cabe no modelo de turn de hoje.**~~ **RESOLVIDO (Loop 3-A, `docs/revamp/C3-runtime-followups-design.md` §6a).** Fila assíncrona persistida `PermissionGate`/`permission_queue` (owner-scoped, correlacionada por `PendingPermission::row_id` — não pelo `PermissionRequestId` do harness, que só é único DENTRO de uma sessão). O consumidor de mode 3 (`spawn_delegated_task_consumer`, um tokio task independente que nunca segura `&mut agent`) agora PAUSA de verdade (`wait_for_permission_resolution`) até: uma decisão explícita de um turno posterior (`AgentLoop::respond_permission`), o timeout (`expires_at` → `Deny{Turn}` fail-closed, registrado na fila), ou a task ser cancelada enquanto pausada (idem). Mode 2 (`run_runtime_backed_turn`) permanece deny-imediato por design — pausar ali seguraria `&mut agent` e congelaria o daemon pra todos os outros owners; documentado no rustdoc de ambas as funções. Testado completo no FakeRuntime (`tests/agent_runtime_cross_turn_permission.rs`, 3 cenários: allow/deny explícito/timeout) + 6 testes unitários do gate. Matriz de suporte por adapter (honesta): nem Codex nem acpx têm pausa longa comprovada em produção — a infra funciona plenamente com qualquer harness que exponha isso; nos dois adapters shipped hoje, o comportamento efetivo permanece `Deny{Turn}` no timeout configurado (10min default), não uma pausa indefinida real-world verificada.
   - **(b) `AgentRuntime::resume` não cobre task-in-flight cross-restart.** Nem `codex_app_server` nem `acpx` fazem replay/buffer de eventos de uma task que já estava rodando quando a conexão caiu — `resume()` reata a SESSÃO, não a TASK. `resume_delegated_task` é honesto sobre isso (submete uma nova task de follow-up) — mas isso significa que hoje **não existe** "a mesma tarefa continua de onde parou" através de um restart real; só existe "a mesma sessão pode receber uma tarefa nova". Se o produto algum dia precisar de continuidade de task genuína, é um furo de protocolo a resolver com os mantenedores do harness, não algo que Bastion pode simular sozinho. **Fora de escopo do Loop 3-A** — furo de protocolo do harness, não um gap do kernel.
   - ~~**(c) `scripts/dump-public-api.sh` tem um ponto cego pré-existente: `pub async fn`.**~~ **RESOLVIDO (Loop 3-A, commit 6c).** Regex ampliado pra `pub\s+(?:(?:async|unsafe|const)\s+)*(fn|...)` — 90 itens `pub async fn` previamente invisíveis apareceram nos baselines regenerados (7 crates: bastion-runtime +30, bastion-cognition +17, bastion-agent-runtime +15, bastion +12, bastion-mcp +9, bastion-mesh +4, bastion-personas +3). Não é regressão — a superfície já existia, só ficou visível ao gate `--check`.
   - ~~**(d) `pending_tx`/`pending_rx` (seam PROACT-05) não é owner-roteado.**~~ **RESOLVIDO (Loop 3-A, `docs/revamp/C3-runtime-followups-design.md` §6d).** Item enfileirado agora é `PendingItem { owner: Option<String>, text: String }`; `spawn_delegated_task_consumer` e `CronService::run_heartbeat`/`on_event` tageiam com o owner real; consumidor (`main.rs`'s `pending_rx` arm) roteia via `run_turn_for(&item.text, owner)`, com `DEFAULT_OWNER` só como fallback pra item sem owner (nenhum produtor atual deixa de nomear um). Testado (`delegate_task_pending_item_never_crosses_owner_boundary`, `crates/bastion-runtime/src/agent/loop_.rs`): dois owners delegam tasks concorrentes no MESMO `AgentLoop`, cada resultado chega tageado com o owner correto, nunca cruzado.

- **`feat(c3): backend selection UX + versioned support matrix`** — `/backends` (lista todo backend conhecido + saúde ao vivo + coverage + auth resolvido) e `/backend use <id>|model|task:<id>|task:none` (troca ao vivo, sem restart, fail-closed — id não resolvível NUNCA aplica troca parcial nem cai silenciosamente pra `model`) via o MESMO mecanismo de cockpit command que `/goals`/`/drift`/`/memories` já usam — reuso, não superfície nova. `docs/SUPPORT-MATRIX.md` (público, versionado): tabela alvo × auth × capabilities × policy-coverage, derivada dos `RuntimeDescriptor` reais + placares vivos da A-05, com nota explícita de que limites/ToS pertencem ao adapter/provider, nunca promessa eterna do Core.
